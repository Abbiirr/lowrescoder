# <a name="header"></a><a name="content"></a><a name="xb9e566159e48e8d7c81a3df361c494255048277"></a>Building a High-Performance Interactive CLI (Claude/Codex Style)
## <a name="challenges-with-naive-approaches"></a>Challenges with naive approaches
Claude Code’s official terminal UI was originally built with Ink (Node/React), which **redraws the entire interface on each update**, causing heavy flicker and inefficiency[\[1\]](https://news.ycombinator.com/item?id=46699072#:~:text=The%20problem%20is%20they%20are,and%20redraws%20for%20each%20update). OpenAI’s Codex CLI avoided this by using a native Rust TUI (Ratatui) that updates incrementally and doesn’t suffer the same flicker[\[1\]](https://news.ycombinator.com/item?id=46699072#:~:text=The%20problem%20is%20they%20are,and%20redraws%20for%20each%20update). In general, high-level or interpreted solutions (like JS/Ink or heavy Python UI libraries) struggle to render fast, flicker-free output for a chatty interface. Community discussions note that Python-based TUIs often become a **performance bottleneck** in non-trivial apps – one developer saw a *20× speed boost* after rewriting a status-line tool from Python to Rust[\[2\]](https://news.ycombinator.com/item?id=46699072#:~:text=Even%20my%20CC%20status%20line,it%20from%20python%20to%20rust)[\[3\]](https://news.ycombinator.com/item?id=46699072#:~:text=It%E2%80%99s%20surprising%20how%20quickly%20the,shells%20out%20to%20C%20modules). In short, to beat Claude Code’s performance, you’ll want a compiled, efficient tech stack that can handle frequent screen updates without redrawing everything.
## <a name="high-performance-tui-tech-stack-options"></a>High-Performance TUI Tech Stack Options
To achieve a fast, flicker-free interactive text UI with **scrollback, a fixed input bar, and arrow-key navigation**, consider using a low-level compiled framework instead of Ink or high-overhead libraries. The developer community explicitly recommends using native terminal UI libraries like **Go’s Bubbletea** or **Rust’s Ratatui** for robust, efficient rendering[\[4\]](https://news.ycombinator.com/item?id=46699072#:~:text=exists) (Codex CLI’s smooth performance attests to this). Since you prefer to avoid Rust and JS, excellent alternatives are available in C++ and Go:

- **C++ with FTXUI:** FTXUI is a modern C++ library for building terminal UIs. It’s *cross-platform* (Linux, Windows, macOS) and handles the quirks of each OS’s console for you[\[5\]](https://steamheritages.co.uk/ftxui/#:~:text=Another%20reason%20why%20FTXUI%20is,as%20many%20users%20as%20possible), so your app will behave consistently on Windows and Linux. FTXUI offers a high-level, React-inspired API: you compose your interface from built-in components (text boxes, input fields, menus, etc.), and it takes care of efficient rendering. It supports **interactive widgets** like input boxes, scrollable lists, checkboxes, dropdowns, etc., which update themselves on user input[\[6\]](https://steamheritages.co.uk/ftxui/#:~:text=A%20UI%20is%20not%20very,your%20data%20with%20your%20display). This is ideal for your use-case – you can create a layout with a scrollable output pane and a fixed-position input bar, and FTXUI will efficiently diff and render updates without flicker. Keyboard input (arrow keys, Enter, etc.) and even mouse events are supported out of the box. Because it’s C++ and has no heavy runtime, performance and resource use are excellent (on par with C/ncurses). In summary, FTXUI gives you a **native CLI look** with minimal effort, and its efficient rendering can outperform Claude Code’s Ink-based approach.
- **Go with TUI frameworks (Bubbletea or Tcell/Tview):** Go is another great choice for a high-performance CLI. Go programs compile to a fast native binary, and libraries like *Bubbletea* (by Charmbracelet) or *Tview* (built on the Tcell terminal library) make it easier to build interactive text UIs. **Bubbletea** uses an Elm-style architecture: you model state and view as pure functions, and it handles terminal rendering and input events. It’s quite capable of complex interfaces, and you can implement features like menus with arrow-key selection (Bubbletea’s list component or a custom update loop can handle up/down arrow keys to move a cursor and Enter to select, as in many example apps)[\[7\]](https://dev.to/andyhaskell/processing-user-input-in-bubble-tea-with-a-menu-component-222i#:~:text=options%20%3D%20append%28options%2C%20fmt.Sprintf%28,s)[\[8\]](https://dev.to/andyhaskell/processing-user-input-in-bubble-tea-with-a-menu-component-222i#:~:text=Enter%20fullscreen%20mode%20Exit%20fullscreen,mode). Bubbletea’s maintainers have continually improved its performance (v1.3+ introduced major rendering optimizations to reduce flicker on slower terminals[\[9\]](https://github.com/charmbracelet/bubbletea/issues/850#:~:text=Support%20for%20Synchronized%20Output%20,rendering%20performance%20improvements%20in%20v1)). For a more straightforward approach, **Tview** provides higher-level widgets (e.g. TextView for scrollable text output and InputField for user input). You can layout these widgets in a Flex container to fix the input box at the bottom and let the output text view expand above. Both Bubbletea and Tview sit atop **tcell** (a Go terminal handling library), which works on Windows console and Unix terminals. This means things like color, resizing, and special keys are handled for you. Go’s concurrency is a bonus – you can spawn a goroutine to stream AI output and update the UI in parallel with user input. The result is a snappy, **flicker-free** interface (so long as you use the latest libs and double-buffer updates) that should meet or exceed the performance of Claude Code’s Node.js implementation in both speed and lower resource usage.

*(Note:* *Anthropomorphic’s own developers eventually abandoned their initial Ink/TS renderer in favor of a custom Zig/CPP “OpenTUI” for Claude Code to reduce flicker[*\[10\]*](https://news.ycombinator.com/item?id=46699072#:~:text=Opencode%20uses%20SolidJS%3A) – which underscores the importance of a low-level approach. You likely won’t need to go as far as writing your own engine in Zig, though; the options above are already proven to be efficient.)*
## <a name="implementing-the-desired-features"></a>Implementing the Desired Features
All of the above tech stack options let you recreate the **native CLI-style UX** you want:

- **Fixed Input Bar with Scrollback:** In an interactive TUI, you typically run in an alternate screen buffer where you control the whole layout. For example, Codex CLI launches a full-screen terminal UI (separate from the regular scrollback) for its chat sessions[\[11\]](https://developers.openai.com/codex/cli/features/#:~:text=Running%20in%20interactive%20mode). You can design your layout with a persistent input box at the bottom and an output log area above it. The output area can accumulate conversation history and be made scrollable. In FTXUI or Tview, this is as easy as putting the input component at the bottom of a vertical container and enabling scrolling on the text view. These frameworks handle redrawing new lines in the output without moving the input bar. This yields a chat-like experience in the terminal – the user can scroll up to see history (e.g. with PgUp/PgDn or ↑/↓ if you wire those keys to scroll), and the input field stays anchored.
- **Arrow-Key Menu Selection & Multi-Stage Prompts:** Your CLI assistant may sometimes present multiple options or require confirmation steps (just like Claude Code or Codex do). TUI frameworks have native support for menus and selectable lists. For instance, OpenAI’s codex resume command brings up a list of past sessions where you **highlight an entry with arrow keys and press Enter to select[\[12\]](https://developers.openai.com/codex/cli/features/#:~:text=reopen%20an%20earlier%20thread%20with,same%20repository%20state%20and%20instructions)**. You can implement similar behavior easily. In FTXUI, you might use a Menu or a list component – it will handle highlighting the current selection as the user presses up/down. In Go’s Bubbletea, you can either use the bubbles/list component or manage selection in your update logic (tracking an index and updating it on key presses, as shown in many Bubbletea tutorials). These libraries also support multi-step interactions: you can push a new “view” or state for the next question. For example, you might first show a list of files to apply a change to; once the user selects one, you swap the UI to a confirmation prompt. Frameworks like Tview even allow modal dialogs or pages for such multi-stage flows. The key point is that **arrow navigation and interactive choices are built-in capabilities** of these TUI toolkits – you won’t have to manually decode escape sequences for arrow keys or implement your own flicker-prone selection UI.
- **Parallel Input and Output Rendering:** A responsive UX means the user can see the AI’s answer streaming in and possibly start formulating the next query without lag. The recommended stacks support concurrency. In Python (with prompt\_toolkit) this would be tricky, but in C++ or Go you can use threads or goroutines. For example, you can spawn a thread to receive chunks of AI-generated text and append them to the output component as they arrive, while the main thread runs the input event loop. FTXUI’s ScreenInteractive::Loop can receive events from another thread via posted tasks, and Bubbletea can send messages to its update loop from goroutines (e.g. using tea.Cmd in Bubbletea to handle asynchronous tasks). This means **output can render incrementally** (character-by-character or line-by-line) *without* freezing the input. The user sees text generated in real-time, and the input box remains responsive (you might disable actual submission until generation is done, but the point is the UI doesn’t lock up or flicker). This parallel rendering is exactly how Codex/Claude stream tokens to the terminal in their clients, and with a solid TUI framework you can achieve the same smooth experience.
## <a name="summary"></a>Summary
To build a **high-performance Claude Code/Codex-like CLI**, avoid heavyweight JS or high-level approaches that redraw too much. Instead, leverage a compiled-language TUI library that updates the terminal efficiently. A **C++/FTXUI stack** or **Go with Bubbletea/Tview** are both excellent choices – they offer native speed, cross-platform support, and convenient abstractions for text-based widgets. OpenAI’s Codex CLI demonstrated the benefit of this approach (using a Rust+Ratatui core for speed)[\[1\]](https://news.ycombinator.com/item?id=46699072#:~:text=The%20problem%20is%20they%20are,and%20redraws%20for%20each%20update), and community feedback reinforces that tools like Bubbletea or Ratatui are well-suited for complex interactive consoles[\[4\]](https://news.ycombinator.com/item?id=46699072#:~:text=exists). By using these technologies, you can implement all the desired features – a flicker-free scrollback log, a fixed input bar, interactive option selection with arrow keys, and streaming output – with minimal latency and resource usage. The result will be a **snappy, user-friendly TUI** that feels like a native terminal experience while matching or exceeding Claude Code’s rendering performance. Just as importantly, it will be easier to maintain than a hacky Python prompt solution, since these frameworks handle the low-level terminal intricacies for you. Good luck building your CLI assistant – with the right stack in place, you’ll have a fast and smooth UI that makes AI coding assistance truly enjoyable to use!
