Concern — Severity: **High** (real-use visible regressions).

The user ran `autocode` live and shared a screenshot showing the default-path TUI during a backend-down session. The screenshot exposes runtime bugs that Stage 4 visual-fidelity work has not touched. These should be the first things Entry `1316` reproduces and fixes. In descending priority:

### 1. HIGH — Status bar leading fields render as empty pipes

Evidence (from the live screenshot):
- top status row reads: `|  |  | ● halted | tasks:0 | agents:0 | q:0 | sandbox:local`
- first three pipe-delimited chunks are visually empty

Code pointer:
- `autocode/rtui/src/render/view.rs:52-58` — `render_status_bar` renders three leading spans unconditionally:
  ```
  Span::styled(&state.status.model, Style::default().fg(Color::Cyan)),
  Span::raw(" | "),
  Span::styled(&state.status.provider, Style::default().fg(Color::Magenta)),
  Span::raw(" | "),
  Span::styled(&state.status.mode, Style::default().fg(Color::Green)),
  ```
- if any of those strings is empty, the separators still render, producing the `|  |  |` prefix

Two orthogonal questions for Codex's investigation:
- why are `model` / `provider` / `mode` empty on default startup? If they are populated from config, startup is failing to populate them. If they are only populated after first backend response, that is a UX bug — the defaults should be readable from config.
- regardless of the root cause, `render_status_bar` should not render empty styled spans or their separators; add a `push_chunk(label, style, opt_value)` helper that skips when `opt_value.is_none() || is_empty`.

### 2. HIGH — Recovery menu is static text, not arrow-key selectable

Evidence:
- Recovery heading (yellow) followed by:
  `Retry  Inspect  Restore`
  `Rewind  Compact  Planning`
  `Esc stay halted`
- no `●/○` focus glyph, no visible selection, no shared picker overlay

Violation:
- user memory `feedback_arrow_key_pickers.md` durably requires **all TUI option lists** to be arrow-key selectable — text dumps are unacceptable

Reference pattern already in the codebase:
- `render_detail_surface` → `DetailSurface::Restore` at `view.rs:560-575` uses `●` for active + `○` for inactive with a footer key hint
- the recovery screen should use the same pattern (or the shared picker overlay in `render_picker` at `view.rs:467`)

### 3. MEDIUM — Recovery screen leaves most of the content area as empty dark void

Evidence:
- `Recovery` + 3 rows of options occupy maybe 6 rows at the top of the content area
- the remaining ~30 rows down to the composer are pitch-black empty space

Options (any of these reads better than the current dead-air layout):
- vertically center the recovery card inside the content area
- shrink the content area to fit the card and expand the composer
- fill with diagnostic content (last request body, HTTP status, retry count, endpoint)

### 4. MEDIUM — User input echoes without clear send/fail status

Evidence:
- transcript shows `> hello` and `/model` as raw user echoes with no assistant response visible between them and the red `Error: Backend not responding` line
- not clear to the user whether (a) `hello` was sent and silently failed, (b) `/model` would have worked but was blocked by halted state, or (c) the `/model` picker opened and was dismissed by the error

Likely root cause:
- during halted / backend-down state, slash commands may short-circuit in the composer handler without giving the user any feedback at the transcript row

Suggested UX outcome: a dim inline status line attached to each orphaned user echo — e.g. `> hello  ↳ send failed (backend down)` — so the user knows which of their inputs reached the backend and which didn't.

### 5. LOW — Positive: outer `" AutoCode "` title is gone

Confirmed in the screenshot. Entry `1313` structural fidelity work is visible. Keeping the finding as evidence that the shell chrome cleanup has shipped to live product.
