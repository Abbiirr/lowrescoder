## Current compliance audit

### HR-1 — Full-screen rendering: ❌ NOT COMPLIANT

Evidence:
- `autocode/rtui/src/render/view.rs:33` — `let size = shell_area(f.area());` wraps the entire TUI in a subregion.
- `autocode/rtui/src/render/view.rs:1102-1123` — `shell_area()` caps width by terminal size:
  - `area.width >= 150` → width `96`
  - `area.width >= 120` → width `84`
  - `area.width >= 100` → width `80`
  - `area.width >= 90`  → `area.width - 12`
  - else                → `area.width - 2`
- Tests that currently assert the centered-narrow behavior (both must be updated):
  - `view::tests::wide_terminal_shell_area_is_narrower_and_centered` at `view.rs:1628-1635` (asserts `shell.width == 84` at `120 cols`)
  - `view::tests::idle_layout_centers_untitled_shell_on_wide_terminals` at `view.rs:1637` (asserts `leading_spaces >= 16`)
- Live evidence from the user's screenshot: at a wide terminal, the TUI occupied a narrow centered region with large dark margins on both sides.

**This is a direct override of the Entry `1310`/`1313` centered-shell Stage 4 direction.** The user's HR-1 supersedes that approval per `AGENT_COMMUNICATION_RULES.md` "The user always has final authority and can override any agent decision."

### HR-2 — Resize support: ✅ COMPLIANT

Evidence:
- `state/reducer.rs:78-83` — `Event::Resize(w, h)` emits `Effect::ResizePty(max(40, w), max(6, h)) + Effect::Render`.
- `main.rs:266-273` — `Effect::ResizePty` calls `child_guard.resize(PtySizeEllipsis)` to propagate to the backend.
- `backend/process.rs:53-56` — PTY `resize()` forwards through `master.resize(size)`.
- `state/reducer_tests.rs:310, 439` — `resize_clamps_pty_but_keeps_raw_terminal_size` and `resize_effect` tests exist.
- `view.rs:23-31` — small-terminal fallback: renders `"Terminal too small (NxN, needs NxN)"` message when below `MIN_TERMINAL_COLS`/`MIN_TERMINAL_ROWS`.

### HR-3 — Multi-size rendering: ⚠ PARTIAL

Evidence:
- Render code handles sizes adaptively: `shell_area` has 5 width branches; `split_detail_columns` clamps sidebar width.
- **But only ONE size is exercised in PTY smoke:** `autocode/tests/pty/pty_smoke_rust_comprehensive.py:30` hardcodes `COLS, ROWS = 160, 50`.
- No multi-size visual regression test exists.
- Unit-test coverage for layout at different sizes is sparse.

### HR-4 — Native scrollback preserved: ✅ COMPLIANT

Evidence:
- `main.rs:145` — `altscreen = args.iter().any(|a| a == "--altscreen")` — default is **false**.
- `main.rs:153-155` — `EnterAlternateScreen` only executed if `--altscreen` is passed.
- In inline mode (default), the terminal's native scrollback buffer is untouched; pre-launch content and post-launch scroll remain accessible via the terminal emulator's native scroll mechanism (mouse wheel, `Shift+PgUp`, right-click scroll, etc.).

**One subtlety the user should know:** the TUI captures mouse-scroll events to drive its own in-app scrollback (`state.scrollback: VecDeque<String>`, mouse-scroll test at `reducer_tests.rs:397`). While the TUI is focused, mouse-wheel scrolling scrolls the TUI's internal scrollback rather than the terminal's native scrollback. Native-scrollback access during TUI operation typically requires terminal-specific keybinds (e.g., `Shift+PgUp`). This is standard behavior for TUIs running in inline mode and is not a break of HR-4 — but if the user wants mouse-wheel to always drive the terminal's native scrollback, that is a policy decision that would require disabling mouse capture.
